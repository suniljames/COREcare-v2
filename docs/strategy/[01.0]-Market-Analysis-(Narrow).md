## Part 1 — The Direct Opportunity

### Label (canonical, propagates verbatim through downstream docs)

> **Caregiver-Mobile-First Care Ops for Private-Duty SMBs**

Encodes:
- **Buyer segment:** Private-duty SMBs (5–200 ADC, US, for-profit)
- **Workflow center of gravity:** Care ops (= scheduling + visit documentation + care plan)
- **Differentiation axis:** Caregiver-mobile-first UX architecture

### Why this label, and the candidates rejected

| Candidate | Rejected because |
|---|---|
| "AI-Native Operations Platform for Non-Medical Home Care" | AI is aspirational in v2's bundle ("future phases") and incumbents already shipped (Compassus deployed ambient AI to 8,000+ clinicians per Q1'26 update). Writing a check the product cannot cash. |
| "Modern Home Care OS for Private-Duty SMBs" | "Modern" is the weakest possible differentiator linguistically. Every challenger calls itself modern. Defeats the purpose of locking a label. |

The chosen label is the only one grounded in something v2 already commits to in its bundle (the explicit decision heuristic: *"Default to the caregiver's mobile experience — they are the highest-volume, lowest-patience user"*) and that the named incumbents *structurally cannot copy cheaply* — HCHB and WellSky's enterprise architectures are desktop-charting-first and would require multi-year rebuilds to retrofit a mobile-native primary surface.

AI and SMB-friendly economics live as supporting moats in §2.6 — not in the headline.

## Part 2 — The Direct Opportunity, Defined Rigorously

### 2.1 Scope Boundary

#### IN-SCOPE

| Dimension | Commitment |
|---|---|
| **Buyer** | Private-duty / non-medical home care agencies, US, for-profit (82% of market per source §2), 5–200 ADC. Single-state and small multi-state operators. Includes single-caregiver agencies through ~200-caregiver agencies per v2 heuristic. |
| **Service mix** | Personal care, companionship, homemaker, ADL support; non-Medicare-billed clinical visits (RN/LPN/CNA/HHA) where the agency does both. State HCBS waiver work permitted *only* where EVV is satisfied by a separate state aggregator (COREcare is back-office, not EVV system). |
| **Workflows** | Shift management (scheduling, assignment, handoff); visit documentation (ADLs, non-clinical vitals, tasks, structured notes); caregiver-mobile-first UX; structured care plans (non-OASIS); family portal as table-stakes baseline (read-only / limited-write); in-app + email notifications; multi-tenant agency admin + super-admin. |
| **Geography** | US only. |
| **Compliance** | HIPAA from day one; multi-tenant RLS; Clerk RBAC (super-admin / admin / manager / caregiver / family); BAAs with subprocessors. |

#### OUT-OF-SCOPE (with NO-GO justifications)

| Segment | Status | Why v2 cannot or should not chase it |
|---|---|---|
| **Enterprise Medicare-certified skilled HH** (HCHB / WellSky / MatrixCare lane) | NO-GO | Requires OASIS-E1 (~100 items, all-payer expansion July 2025), PDGM 30-day episode billing, HHVBP — none in v2 domain model. Buyer = 500+ ADC enterprise (8% of agencies / 45% of patients per source §3) with $50K–500K+ ACVs and 6–18 mo implementations. v2 stack (single Mac Mini Docker Compose, autonomous AI-driven workflow, no formal sprint process) cannot service enterprise sales motion. KLAS-tier certification is years of work. |
| **Pure RCM / billing-as-a-service** | NO-GO | WellSky RCM ($11B+ Medicare claims), Waystar ($1.8T annually), ABILITY/Inovalon (98% first-pass) — entrenched at unattainable scale. % of collections (3–8%) revenue model is services-heavy, clashes with v2's product-led SaaS posture. |
| **Point-solution scheduling only** | NO-GO | Source §7 lists "pure scheduling without clinical integration" under weak moats. Head-to-head with ShiftCare ($8/user) and AxisCare ($200/mo) on price/scheduling alone is a losing axis. Bundling visit docs + family + care plan is what creates the stickiness. |
| **EVV aggregator / state Medicaid gateway** | NO-GO (direct); adjacency | HHAeXchange owns state-aggregator moat. v2 anti-pattern explicitly prohibits real-time caregiver GPS — directly inconsistent with EVV core req. Moves to Part 3. |
| **RPM device platform** | NO-GO | Different category (12% market / 28.5% CAGR), device-centric (Biofourmis / TytoCare / 100Plus), FDA-cleared hardware = capital-intensive. v2 has no device strategy. |
| **Hospice-specialized clinical** | NO-GO (direct); adjacency | Distinct regulatory regime (Medicare hospice benefit, IDG meetings, bereavement). Dominated by HCHB Hospice and Netsmart myUnity Hospice. |
| **SNF / post-acute (LTPAC)** | NO-GO | PointClickCare 70% share = structural lock per source §9. Different setting, MDS not OASIS, longer LOS. |
| **Senior living resident engagement** | NO-GO (direct); adjacency | Caremerge/Icon, Aaniie active. Buyer is community operator, not agency. |
| **Hospital-at-Home / high-acuity** | NO-GO | DispatchHealth + Medically Home dominate; CMS waiver extended to 2030 per Q1'26 update raises stakes. Acute clinical workflows + health-system buyer outside v2. |
| **Employer-paid family caregiver benefit (B2B2C)** | NO-GO (direct); adjacency | Cariloop / Carallel / Careforth — sold to employers and health plans, not agencies. Different buyer entirely. |
| **Caregiver gig marketplace** | NO-GO | ShiftKey / CareRev model. Source notes "race to the bottom on wages" backlash. v2 anti-pattern: "no social features or inter-agency communication." |
| **International (UK / CA / AU NDIS)** | NO-GO (direct); adjacency | Distinct regulatory + funding models. v2 stack is US-English / US-compliance only. |

### 2.2 TAM / SAM / SOM (re-derived)

Not reusing the $4.5–17.7B headline. Built from source §2, §3, §4.

#### TAM (lane)

US private-duty home care software, scheduling-and-workforce + care-coordination subcategories combined, non-Medicare slice.

- Mordor narrow software TAM: $4.51B (2025) → $8.20B (2030) at 13% CAGR
- Scheduling & Workforce subcategory: 25% = **$1.13B** (source §3)
- Care Coordination subcategory: 5.5% = **$248M** (source §3)
- Combined "care ops" = **$1.38B**
- Apply non-Medicare-certified filter: 20,000 of ~33,200 US agencies are non-Medicare per source §3 ≈ 60%
- **Lane TAM ≈ $830M (2025) → ~$1.5B (2030)** [the 60% allocation to non-Medicare is my derivation — source doesn't split subcategory revenue by payer mix; flag as **[unverified]**]

#### SAM

US private-duty agencies in the 5–200 ADC band.

- 20,000+ private-duty / personal-care agencies (source §3)
- 70% SMB (<100 ADC) + ~half of the 22% mid-market band (the 100–200 ADC slice) ≈ **81% of agencies**
- = **~16,200 agency buyers in band**
- Realistic ARR per agency for this tier: **$15K–$25K** (anchored to AxisCare $200–$500/mo tiers + Alora $295/mo + WellSky Personal Care $100/client × small client counts; consistent with source §4 "$15K–150K annually" and "1–3% of revenue")
- Midpoint $20K × 16,200 agencies = **SAM ≈ $324M ARR**

#### SOM (5-yr, per source §2 "5–10% market share")

- 5% of SAM logos = 810 agencies; 10% = 1,620 agencies
- At $20K ARR midpoint: **$16M–$32M ARR steady-state**
- **12–18 month sellable wedge:** 30–100 paying agencies = **$600K–$2M ARR** **[unverified]** — derived from typical SMB SaaS year-1 acquisition curves; source has no comparable benchmark.

### 2.6 Differentiation Thesis

Honest split between *moat*, *credibility advantage*, and *parity*.

| Axis | Type | Rationale |
|---|---|---|
| **1. Caregiver-mobile-first UX architecture** | **Real moat (18–36 months)** | HCHB / WellSky / Netsmart have desktop-charting-first architectures; mobile apps were grafted on later. AxisCare and AlayaCare have mobile but workflow-design center is web. v2 can build the entire visit-and-documentation primary surface as mobile-native. Retrofitting this is a multi-year rebuild for incumbents — they won't do it cheaply. |
| **2. RLS-enforced multi-tenancy + Clerk RBAC with family as first-class role** | **Credibility advantage, not moat** | Architecturally cleaner than the looser permission models on AxisCare / CareSmartz360 / Alora. Sells well in HIPAA conversations and to agencies preparing for state surveys or LTC-insurance audits. Matchable in 12–18 months by any motivated competitor. |
| **3. AI documentation with PHI redaction + Anthropic SDK** | **Parity (currently behind)** | Source + Q1'26 update show this crossed from pilot to production in 2025–2026 (Compassus 8K clinicians). v2 bundle puts Claude API in "future phases" — v2 is *behind* on this axis today. Differentiation will come from *implementation discipline* (PHI redaction, audit logging, per-task model selection) rather than from having AI at all. |
| **4. SMB-friendly economics from a modern stack** | **Real moat vs enterprise; parity vs ShiftCare/AxisCare** | FastAPI + single-Postgres + Docker run-cost is dramatically below WellSky-tier infra. Translates to a sustainable price floor. But ShiftCare ($8/user) and AxisCare ($200/mo) already price aggressively — v2 doesn't undercut, it matches with better unit economics behind the scenes. |
| **5. Audit-first / privacy-by-default product DNA** | **Credibility advantage** | v2 bundle hard rules (RLS-only, PHI exclusion from logs, structured audit logging, no PHI in notifications, BAA tracking) read as engineered-in, not bolted-on. Rare in SMB-tier vendors. Sellable but matchable. |

**Honest assessment:** Only **#1** is a durable moat. **#4** is a moat against enterprise but not against same-tier competitors. **#2, #3, #5** are credibility/execution advantages that any motivated competitor can match within 12–18 months. The headline pitch should lead with caregiver mobile experience, with audit-and-privacy and SMB economics as supporting proof.

## Part 3 — Adjacencies as Future Scope

These are **future optionality, not roadmap**. Sequencing belongs to downstream planning docs. The architectural-implications-NOW column is the load-bearing one.

### 3.1 IDD / HCBS Waiver Providers

- **Brief:** Tech for agencies serving Medicaid HCBS waiver beneficiaries with intellectual / developmental disabilities. 15,000+ providers per source §3.
- **Why adjacent:** Same agency-tenant model; same Caregiver / Client / Visit / Shift entities; shift-based work; often overlapping personnel (HHA / CNA also work IDD).
- **Additive:** State EVV aggregator integrations (HHAeXchange, Sandata, state-built), service-authorization tracking against Medicaid units, ISP (Individual Support Plan) format vs. care plan, billing in 15-minute units, state-specific incident reporting.
- **Architectural implications NOW:**
  - Care Plan entity needs polymorphism or extensible attributes — don't lock to one rigid schema
  - Visit needs an extensible verification field — don't assume a single clock-in mechanism
  - Tenant model needs to support state-context (a multi-state agency operates under different rules per state)
  - Audit log granularity must support state-level Medicaid audits (RAC requests are routine)
- **Disqualifying signals:** state EVV aggregator monopolies tighten such that COREcare can't push verified visits in (e.g., HHAeXchange requires data exclusivity); CMS pulls waiver flexibility; v2 commits to a single rigid Care Plan schema that can't represent ISPs.

### 3.2 Pediatric Home Care

- **Brief:** Private-duty pediatric nursing for medically complex kids. KanTime is incumbent (source §11, "private duty, pediatric").
- **Why adjacent:** Same Agency / Caregiver / Client / Visit / Shift entities; family-portal-heavy because parents are primary stakeholders.
- **Additive:** Parent-as-caregiver workflows, school / IEP coordination, longer continuous shifts (8–12hr), G-tube / vent / trach documentation, pediatric medication dosing, Medicaid managed care contracts, heightened background-check stringency.
- **Architectural implications NOW:**
  - Client entity needs minor flag + guardian relationship, not just "Family Member"
  - Family Member role needs the option to be the *responsible decision-maker*, not just informed observer
  - Longer-shift modeling: Visit / Shift schemas should not assume short duration
  - Consent / data-sharing for minors hits 42 CFR Part 2, COPPA for portal users <13, and state-specific minor-consent laws
- **Disqualifying signals:** Aveanna / Family First and similar consolidate the pediatric tier (per Q1'26, Aveanna acquired Family First); Medicaid pediatric reimbursement crisis forces agency exits; state Medicaid managed care plans mandate proprietary platforms.

### 3.3 Hospice

- **Brief:** End-of-life care; 5,800+ Medicare-certified hospice agencies (source §3).
- **Why adjacent:** Caregiver / Visit / Shift transfer; Care Plan transfers conceptually but becomes Plan of Care under hospice rules; Family Member becomes critically central (bereavement extends post-death).
- **Additive:** Medicare hospice benefit billing, IDG meetings every 15 days, level-of-care escalation (routine → continuous → general inpatient → respite), bereavement program (13 months post-death), volunteer management (5% requirement), Medicare hospice cap calculations.
- **Architectural implications NOW:**
  - Family Member entity should support post-client-death state — don't tie family access strictly to active client status
  - Care Plan needs versioning + IDG-cycle metadata
  - Audit log must retain records 7 years post-death per Medicare
  - Notification rules need end-of-life context awareness (when *not* to send routine reminders)
- **Disqualifying signals:** HCHB Hospice and Netsmart myUnity Hospice deepen integration moats faster than v2 could close; Medicare hospice cap reform forces consolidation among small operators.

### 3.4 Adult Day Services *[flagged: weakest fit in set]*

- **Brief:** Facility-based daytime care for elderly or disabled adults.
- **Why adjacent:** Client / Caregiver / Family Member transfer. The *least* adjacent in this set because center of gravity moves from in-home Visit to facility-based group programming.
- **Additive:** Facility schedule (rooms, group activities), transportation coordination, facility licensure tracking, group billing units, often Medicaid waiver-funded.
- **Architectural implications NOW:**
  - Visit assumes home-based location — would need a Site / Location entity
  - Group-care semantics conflict with 1:1 Visit model
  - Transportation = third-party driver / vehicle entities not in v2 today
- **Disqualifying signals:** Most operators bundle adult day with senior living or hospice; persistent COVID-era attendance decline never recovers; unclear whether the segment justifies a software-vendor business.
- **Honest note:** v2 should **not** invest architectural flexibility solely for this — it's downstream of the senior living adjacency. Including for prompt completeness.

### 3.5 Senior Living Resident / Family Engagement

- **Brief:** Software for assisted-living / independent-living operators to engage residents and families. Caremerge / Icon, Aaniie active per source §11.
- **Why adjacent:** Family Member entity transfers directly; Care Plan becomes Service Plan; multi-tenant (community = tenant) maps cleanly.
- **Additive:** Community calendar / activities, dining preferences, resident directory, family billing portal, integration with Yardi / RealPage property management, building access systems, Alexa / voice integrations (Caremerge precedent), wellness check-ins.
- **Architectural implications NOW:**
  - Tenant entity should not hardcode "Agency" — keep tenant naming abstract enough to also mean "Community"
  - Client should be representable as a Resident (long-tenure, low-acuity-change) without schema gymnastics
  - Shift / Visit semantics break for staffing in a building — would need a parallel "Staff Assignment" model
- **Disqualifying signals:** PointClickCare extends senior-living dominance from SNF; Caremerge / Icon and Aaniie consolidate the family-engagement tier; v2 hardcodes "Agency" in too many places.

### 3.6 Behavioral Health HCBS Overlap

- **Brief:** HCBS-funded behavioral health support (community-based MH / SUD services, peer support, ACT teams).
- **Why adjacent:** Agency / Caregiver / Visit transfer; service-authorization model overlaps with #1 IDD; documentation overlaps with HCBS care plan.
- **Additive:** SAMHSA-aligned documentation, **42 CFR Part 2 substance-use confidentiality (much stricter than HIPAA)**, CCBHC certification workflows, peer-support credential tracking, crisis-response logging, treatment-plan goals tied to outcome measures (PHQ-9, GAD-7).
- **Architectural implications NOW:**
  - PHI redaction and audit logging must extend to handle 42 CFR Part 2's heightened restrictions (consent-per-disclosure model) — bake the consent model in now, don't retrofit
  - Care Plan needs measurable-outcomes / scores schema
  - Notification rules must support "do-not-contact" + crisis flags
- **Disqualifying signals:** Netsmart deepens behavioral-health vertical lock; CCBHC funding model contracts; v2's PHI redaction stays HIPAA-only and can't meet 42 CFR Part 2 stringency.

### 3.7 Employer-Paid Family Caregiver Support

- **Brief:** B2B2C platform sold to employers / health plans to support employee-caregivers (Cariloop, Carallel, Careforth per source §11).
- **Why adjacent:** Family Member persona transfers but becomes the *primary* user; Care Plan becomes care-coordination guide rather than service contract.
- **Additive:** Employer benefits integration (Workday / ADP), care-advocate human-services layer, eligibility verification, provider-directory + matching (vs. employed caregivers), respite-care payment infrastructure, content library.
- **Architectural implications NOW:**
  - Auth model assumes Clerk-org-tenant = agency; B2B2C requires *employer-tenant* with families as members. Keep the tenancy abstraction loose
  - Family Member role needs upgrade path to "primary user" not just "viewer"
  - AI agent foundation should be designed so Claude-assisted care navigation can serve a family member directly, not only an admin
- **Disqualifying signals:** Employer benefits market consolidates around 2–3 winners (already advanced); health-plan vertical integration absorbs the category; v2 family portal stays read-only-by-architecture.

### 3.8 Credentialing-as-a-Service Spinout

- **Brief:** Standalone caregiver credential repository portable across employers (license, training, background check, BLS, immunization records). CareAcademy-adjacent but caregiver-owned rather than agency-owned.
- **Why adjacent:** Caregiver entity becomes the center; agency becomes one of many consumers of a caregiver-owned record.
- **Additive:** State license verification API integrations (NURSYS for nursing), training / CEU tracking with state-board mapping, OIG / SAM exclusion checks, background-check provider integrations (Checkr / Sterling), caregiver-owned identity layer (different auth model than employer-tenant).
- **Architectural implications NOW:**
  - Caregiver should not be modeled as a hard child of Agency — should be a portable identity that *associates with* one or more Agencies via a join table
  - Credential data should not be stored only inside agency-RLS scope — needs a caregiver-scoped data layer that persists across employer changes
  - Clerk auth model needs to support caregiver-as-primary-account, not just caregiver-as-agency-employee
- **Disqualifying signals:** CareAcademy or another incumbent wins state-board partnerships that establish official credential repositories; states mandate credential systems through aggregators.

### 3.9 International — AU NDIS, Canada

- **Brief:** NDIS provider software (AU disability scheme, ShiftCare's home market) and Canadian home care (AlayaCare's home market) per source §9.
- **Why adjacent:** Same Agency / Caregiver / Client / Visit / Shift core; same family-portal value; same caregiver-retention pain.
- **Additive:** NDIS-specific funding line items, plan-management workflows, NDIS Quality & Safeguards Commission compliance, AU privacy law (Privacy Act 1988 + APPs), Canadian PHIPA / PIPEDA compliance, French-language support (Quebec), payroll integrations (Xero AU, QuickBooks CA).
- **Architectural implications NOW:**
  - i18n from day one in UI strings — even if only English ships, don't hardcode strings
  - Date / time / currency formatting via locale-aware utilities
  - PHI / privacy abstractions parameterizable per jurisdiction (HIPAA hardcoded = can't be retrofitted to PHIPA cleanly)
  - Tenant configuration should support per-tenant jurisdiction setting
- **Disqualifying signals:** ShiftCare locks in AU NDIS market further; AlayaCare consolidates Canada; AU / CA regulatory regime tightens such that local incorporation is mandated (data-residency requirements that defeat US-hosted SaaS).

### 3.10 Hospital-at-Home Enabler *[added]*

- **Brief:** Agency-side coordination layer for high-acuity HaH programs (DispatchHealth, Medically Home, health-system-run programs). **CMS waiver extended to 2030 per Q1'26 update — single biggest investment overhang lifted.** v2's role would be the home-side coordination, not the acute clinical platform.
- **Why adjacent:** Caregiver / Visit / Shift core transfers; v2 handles home-side coordination while leaving acute clinical to specialists.
- **Additive:** HL7 / FHIR integration with hospital EHRs (Epic, Cerner), clinical-escalation workflows, biometric device data ingestion, pharmacy integration, on-call clinician routing, post-discharge transition handoff.
- **Architectural implications NOW:**
  - Visit schema should accept *external clinical events* (not just caregiver-recorded) — keep it event-sourced or extensible
  - Integration layer should be designed as a first-class concern, not a bolt-on (FastAPI's clean separation supports this — *don't* let business logic leak into the integration boundary)
- **Disqualifying signals:** DispatchHealth / Medically Home extend their proprietary stack down to the agency side; CMS reverses the waiver again (looks unlikely post-2030 extension); health systems vertically integrate HaH and only buy from EHR vendors.

### 3.11 VBC for HCBS *[added]*

- **Brief:** Value-based contracting infrastructure for HCBS providers — performance-tied payments based on outcomes (hospitalization avoidance, ED diversion, fall prevention).
- **Why adjacent:** Agency / Client / Visit transfers; care plan becomes outcome-tracked; family portal becomes a member-engagement surface.
- **Additive:** Outcome-measure library (ED visits, hospitalizations, ADL decline, fall events), risk stratification, payer contract management, attribution logic, claims data ingestion from payers, quarterly performance reporting.
- **Architectural implications NOW:**
  - Visit data should be queryable by *episode* and *time-window*, not just by visit-id — design indexes accordingly
  - Audit log must retain longitudinal change history of Care Plan goals (proving what was tracked when matters)
  - PHI sharing with payers needs explicit consent + minimum-necessary enforcement at the *field* level, not just row-level
- **Disqualifying signals:** VBC adoption in HCBS stalls (Medicaid managed-care plans don't push it down to private-duty agencies); CMS HHVBP framework expands to non-Medicare HCBS in a form that mandates HHAeXchange-style aggregator submission.

### 3.12 Caregiver Retention Overlay *[added]*

- **Brief:** Productized retention / engagement layer addressing 79% turnover (#1 industry pain per source §1). CareAcademy-adjacent but agency-bundled rather than standalone training.
- **Why adjacent:** Caregiver entity already in v2; everything new is workflow + content.
- **Additive:** Shift-preference matching, peer recognition / kudos, pay-stub visibility, training assignment + completion tracking, quick-pay / earned-wage-access integrations (DailyPay / Tapcheck), exit-intent surveys, longitudinal engagement scoring.
- **Architectural implications NOW:**
  - Caregiver entity should track engagement signals (sessions, response time, app opens) — instrument early or backfill blind
  - Notifications need to support celebratory / positive payloads, not only operational ones
  - AI agent foundation should be available to suggest retention interventions ("Caregiver X has missed 3 shifts in 2 weeks — suggest check-in")
- **Disqualifying signals:** CareAcademy or a payroll-integrated player (DailyPay) bundles retention + EWA + training into a category-defining product before v2 reaches scale; agency operators decide turnover is fundamentally a wage problem and tooling can't help.

## Part 3.x — "Don't Foreclose" Checklist

Concrete v2 design decisions to stay mindful of NOW so adjacencies remain reachable. Downstream planning docs should consult this most often.

1. **Tenant abstraction.** Don't hardcode "Agency" in tenant naming, configuration, or branding contexts — adjacencies #5 (senior living) and #7 (employer benefit) need Community / Employer.
2. **Caregiver as portable identity.** Model Caregiver as an entity that *associates with* Agency via a join, not as a child of Agency. Preserves credentialing spinout (#8) and multi-employer caregiver workflows.
3. **Care Plan extensibility.** Don't lock Care Plan to a single rigid schema. Use polymorphic types or extensible attribute storage to accommodate ISPs (#1), pediatric care plans (#2), hospice POCs (#3), VBC outcome plans (#11), behavioral-health treatment plans (#6).
4. **Visit verification mechanism behind an interface.** Keep clock-in/out abstract — signature, photo, Wi-Fi, NFC, or future EVV-aggregator handoff (#1, #10) should all plug in.
5. **Visit as event-sourced or event-extensible.** Allow external clinical events to attach to a Visit (HaH integrations #10, payer claims data #11, biometric device feeds).
6. **Family Member role with upgrade path.** Don't permanently brand Family Member as read-only / secondary. #2 (parent-decision-maker), #7 (employer benefit primary user), #3 (post-death bereavement contact) require role flexibility.
7. **Tenant-scoped jurisdiction setting.** Tenant config should support a jurisdiction parameter from the start (state for US, country for international #9) so compliance rules can be parameterized rather than hardcoded.
8. **PHI redaction parameterizable per jurisdiction / regulation.** Don't hardcode HIPAA. Behavioral health 42 CFR Part 2 (#6), AU Privacy Act (#9), Canadian PHIPA (#9), pediatric COPPA (#2) all require different rules.
9. **Consent model at field-level.** Build consent tracking with field- / disclosure-level granularity (not just row- / role-level). 42 CFR Part 2 (#6) and VBC payer-data sharing (#11) demand it.
10. **Audit log retention longevity.** Default to ≥7-year retention with archival path. Hospice (#3), VBC (#11), and Medicaid audit defense (#1) need long horizons.
11. **i18n from day one.** Externalize UI strings, use locale-aware date / time / currency formatting — even if only en-US ships now (#9).
12. **Integration boundary as first-class concern.** FastAPI service-layer cleanliness must extend to a real integration layer (HL7 / FHIR for #10, payroll / QB for #12, payer data #11, EVV aggregator #1). Don't let integration logic leak into routers or services.
13. **Notification rules engine, not hardcoded notifications.** Adjacencies vary wildly in notification semantics (do-not-contact for behavioral health #6, celebratory for retention #12, suppression around end-of-life #3, employer-routed for #7). Build a rules layer, not if/else.
14. **Caregiver engagement instrumentation.** Track caregiver app sessions, response latency, opens — adjacency #12 needs this longitudinally; backfilling is impossible.
15. **AI agent boundary serving multiple personas.** Claude API integration should not assume the consumer is admin / clinician. #7 (family-direct AI) and #12 (manager-suggesting AI) want different consumers.
16. **Audit log change-history granularity.** Not just access logging — *change* logging on Care Plan goals and Service Plan items, with longitudinal queryability. VBC (#11) and survey defense (#1) require it.
